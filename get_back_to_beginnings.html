Whose code is that?

It's always belongs to whole TEAM

Leave the code cleaner than you found it

It's always teams responsibility when something goes wrong. With code is the same. Last team member who was working on some part of the should fix everything that should be fixed That change doesn't mean big change always - that can be smaller fix - variable name, extract some functions... whatever - just improve your code. Imagine the got gets better as times passes.

Bad code

Treat your code as valuable member of the team. Like member of the management can screw up - code can do it also if it is messy. Hiring an army of developer can't save you from bad code. Try to write it as good as possible. Do not say to yourself I will do it next time - LeBlanc's law says - Later equals never. Bad code is your mistake. Not managers - only yours. It's your job to defend your code - like doctor who is washing his hands before surgery. It takes time but it is your duty. Deadline is not excuse. If you make mess you will never reach that deadline. The only way to go fast is to keep code clean as much as possible all the time. Imagine you code is building. What happens with building when you stop caring about it. The same happens to the code.

Bad code properties

Has tendency to be even worse

Without formating

Tries to do everything

Untestable

Hard dependencies

Code is getting worse as time passes. We should not wait - fix it all the time. That's the only way you can prevent degradation.

Kent Beck's rules of simple code

Runs all the tests

Contains no duplication

Express all designs ideas that are in system

Minimizes the number of entities such as classes, methods, functions, and the like

Testable

Readable

Elegant - well designed

Readable
Read 10x more than write. We are constantly reading old code. If you want to write code more quickly that means that you have to make it more readable. Easy to read is easy to write.

How to write good code

Knowledge

Hard work

Knowledge
You have to learn basics of software architecture and design patterns, follow good good practice and principles. You have to read a lot, you have to read every day. Follow people with high knowledge, listen to them, read their books. Go to conferences, speak about programming with your colleagues. Do it as often as it possible.

Hard work
It's like learning to ride bicycle from the book or learning to fight watching Bruce Lee movies. It's impossible to learn to code just from to book. You learn all the time. As Uncle Bob Martin said in his book "You must sweat over it". You have to write the code, you have to think about it, you have to hate it, hate yourself because you fail. Head up, refactor your code, shine for the moment, enjoy the fact that you are brilliant mastermind and yet again watch yourself failing. Read your code, read other peoples code. Maybe someone is born with code-sense but most of the time it's not. Talented football player is not the best football player. The best player is one who trained a lot every day.

Global State

Set data-state="something" on a slide and "something" will be added as a class to the document element when the slide is open. This lets you apply broader style changes, like switching the page background.

State Events

Additionally custom events can be triggered on a per slide basis by binding to the data-state name. 


Reveal.addEventListener( 'customevent', function() {
	console.log( '"customevent" has fired' );
} );
