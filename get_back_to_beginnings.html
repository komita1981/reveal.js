Readable
Read 10x more than write. We are constantly reading old code. If you want to write code more quickly that means that you have to make it more readable. Easy to read is easy to write. If your code is well design, clean and elegant than everyone will take care of it also. No one will try to make mess in that code. Tell story about fire in beautiful house

How to write good code

Knowledge

Hard work

Knowledge
You have to learn basics of software architecture and design patterns, follow good good practice and principles. You have to read a lot, you have to read every day. Follow people with high knowledge, listen to them, read their books. Go to conferences, speak about programming with your colleagues. Do it as often as it possible.

Hard work
It's like learning to ride bicycle from the book or learning to fight watching Bruce Lee movies. It's impossible to learn to code just from to book. You learn all the time. As Uncle Bob Martin said in his book "You must sweat over it". You have to write the code, you have to think about it, you have to hate it, hate yourself because you fail. Head up, refactor your code, shine for the moment, enjoy the fact that you are brilliant mastermind and yet again watch yourself failing. Read your code, read other peoples code. Maybe someone is born with code-sense but most of the time it's not. Talented football player is not the best football player. The best player is one who trained a lot every day.

Numbers as part of variable name (no a1, a2,...)

Words like Manager, Info, Data

Functions

Basic line of organization

Functions are verbs of every programming language and the classes are nouns (Clean Code) Think about system as stories to be written. Functions need to fit cleanly together into a clear and precise language to help you with that telling (Clean Code) find more definition for this section

Functions should

be small

do one thing

have descriptive name

have up to 3 arguments

not contain more than one or two indents

First is that functions should be small. The second is that should be smaller than that (Clean code) Blocks within if-else and while statement should be one line long (Clean Code) I also prefer this in all other loops Functions should do one thing. They should do it well. They should do it only. (Clean Code) Functions should not have side effects - do in functions only one thing - what name of that function is - no more no less If you use try-catch - than it should be the only one block in function One level abstraction bellow name of the functions You know you are working on clean code when each routine turns out to be pretty much what you expected Long descriptive name is better than short enigmatic name, better than long descriptive comment Choosing good nae is important - remember that you read 10x more than you write The ideal number of params is 0 - niladic functins Next come with 1 param - monadic Followed by 2 - dyadic 3 - Triadic - should be avoided 4 and more params - Polyadic - Think about rewriting your function (T. Otwel) Try to pass object instead creating polyadic functions Do not use flags in functions - create separate functions instead
