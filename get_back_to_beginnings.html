Clean code

Why should we write it?

Everyone wants to write code that other people can understand

We strive to be better programmer

Kent Beck's rules of simple code

      1. Runs all the tests
      2. Contains no duplication
      3. Express the intent of programmer
      4. Minimizes the number of classes and methods

Given in order of importance

1. Testable system passes all tests and is completely tested. System that are not testable should not be deployed because they can't be verified. Personal experience - when I started writing tests I also started writing better code because I could not test bad code and when you write piece of bad code and code that is not designed properly you will notice it immediately in your tests. Encourage SRP and DIP. "The fact that we have these tests eliminates the fear that cleaning up the code will break it". Good tests are also kind of good documentation
2. Evil of every system. DRY - Dont repeat yourself
3. Code should express all designs ideas that are in system. Think about maintenance - if your code reveal your intent people who work after you on that code will spend less time. Some desing patterns solve this problem - Every time you duplicate your code you should create some kind of abstraction Template, Strategy or Visitor pattern. Use if/else and switch case carefully
4. Minimizes the number of entities such as classes, methods, functions, and the like.

Philosophies for Effective Programming

  1. Always write high quality, clean code
  2. Refactor ruthlessly
  3. Think before you code
  4. Develop in small steps
  5. Write understandable code
  6. Adopt and follow coding standards and guidelines
  7. Add documentation to improve code readability
  8. Write documentation before you write the code
  9. Follow the 30 seconds rule
  10. Smaller methods are usually better
  11. Optimize your code only as last resort
30 seconds rule - Another programmer should be able to look at the method, a class or an interface definition. Programmer should understand what it does, why it does it and how it does it in less than 30 seconds. This can be achieved if there is informative header documentation, a name that clearly conveys the intent, by doing what it is supposed to - no more no less.
